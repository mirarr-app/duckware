What it is

A real phone on the other end of every request

Virtual SMS providers are convenient until a region, a carrier, or a compliance rule blocks them. httpSMS takes a different bet: keep a real Android device online, install its companion app, and treat that handset’s SIM as the transport. Your backend speaks a plain HTTP API; the phone does the actual send and receive.

The hosted path at httpsms.com covers the common case — API key, push notification to the app, delivery reports — while the open-source monorepo (API, web dashboard, Android client, Docker Compose) lets you run the same stack on your own infrastructure when data residency or cost matters more than convenience.

How a message moves

Accept fast, deliver on the device

Outbound traffic is asynchronous by design. You POST /v1/messages/send with an x-api-key, get a quick acceptance, and the platform schedules a push to the Android app. The phone fetches the payload, fires the native SMS API, then reports send and delivery status back. Inbound texts reverse the path: the app captures the message and forwards it to the webhook you registered.

Authenticate

API keys live in the dashboard settings and ride every request as a header.

Queue & push

The API accepts the message, then notifies the paired Android app over push.

Send on-device

The app uses the phone’s SMS stack — including dual-SIM selection when needed.

Webhook back

Inbound SMS and delivery events land on your endpoint for automation.

Why it earns a showcase

Three strengths that hold up under scrutiny

Carrier-native reach

Because messages leave a real SIM, you keep the routing and reputation of a consumer line — useful for locales where rented virtual numbers fail silently.

Developer-shaped surface

REST endpoints, SDKs (Go, JS, and friends), scheduled sends, optional E2E encryption, and MMS attachments make it feel like infrastructure, not a hobby script.

Self-host when you must

Docker Compose boots the API and web UI against your own Firebase and SMTP; the Android APK pairs to that private control plane instead of the public cloud.

Audit notes

Who this is for — and what to watch

httpSMS fits teams that already operate (or can operate) an Android device with a working SMS plan: OTP pipelines in stubborn markets, two-way support threads that must look like ordinary texts, or homelab automations that refuse to rent a Twilio number. Pair the phone with a stable power and data setup; the gateway is only as reliable as the handset staying online.

Treat rate limits, carrier spam filters, and local telecom rules as first-class constraints — the API will not paper over them. For production, prefer the documented webhook events for inbound and delivery state rather than polling, keep API keys rotated from the settings panel, and if you self-host, budget time for Firebase credentials and Turnstile keys before docker compose up.
