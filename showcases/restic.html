restic

Encrypted, deduplicated backups that treat restore as seriously as save.

Read Documentation View Source

Typical Workflow

bash
restic init --repo sftp:user@backuphost:/data
restic -r sftp:user@backuphost:/data backup ~/work
snapshot 40dc1520 saved · 1.582 GiB in 0:29

What is restic?

Most backup tools either feel like an afterthought or demand a dedicated server stack. restic is a single Go binary that runs on Linux, macOS, Windows, and the BSDs — no daemon, no agent fleet, no proprietary cloud lock-in.

You point it at a repository (local disk, SFTP, S3, Backblaze B2, Azure, GCS, or anything rclone supports), take encrypted snapshots on a schedule, and restore individual files or entire trees when disaster strikes. The design philosophy is blunt: backups should be frictionless enough to run daily, and restores should be provable before you need them.

Architectural Highlights & Strengths

  • Encryption Against Untrusted Storage

    Every blob is encrypted client-side before it leaves your machine. restic assumes the remote backend is hostile — shared hosting, cloud buckets, or admin-accessible servers — and uses cryptography to guarantee confidentiality and integrity. Your password is the only key; lose it and the data is gone, by design.

  • Incremental Snapshots with Deduplication

    Content-defined chunking splits files into variable-size pieces so that even small edits do not re-upload entire files. Duplicate chunks are stored once across snapshots, keeping repository growth proportional to actual change — not total data volume. Restores pull only the blocks needed for the files you request.

  • Backend-Agnostic by Design

    Native support covers local paths, SFTP, REST servers, Amazon S3, MinIO, OpenStack Swift, Backblaze B2, Azure Blob, and Google Cloud Storage. Anything else routes through rclone. Swap backends without changing your backup workflow — the repository format stays the same.

Restore and Browse

When recovery time matters, restic offers two paths: targeted file restore or FUSE mount to browse snapshot history like a filesystem:

bash
# List snapshots and restore a single file
restic -r /backup snapshots
restic -r /backup restore latest --target /recovery --include report.pdf

# Or mount the repo and browse interactively
restic -r /backup mount /mnt/restic

Verify Before You Need It

A backup you cannot restore is worse than no backup at all. restic ships with integrity checks and reproducible release binaries (since v0.6.1), so you can audit that your repository is healthy and that published builds match the source:

bash
restic -r /backup check
using temporary cache · no errors were found
