Release path

Indexer → filter → client

RSS is too slow for early swarm participation. autobrr sits on IRC announce channels (and Torznab / Newznab / RSS when IRC is not available), evaluates every drop against your rules, then fires the action you configured.

01 · Indexer

Live IRC announces across 75+ trackers, plus RSS, Torznab, and Newznab feeds for indexers without channels — torrents and Usenet in one place.

02 · Filter

Rule sets with RegEx, size, resolution, tags, and more. Matches become actions; noise gets discarded before it ever touches a client.

03 · Client

Push to qBittorrent, Deluge, rTorrent, Transmission, Porla, the *arr suite, SABnzbd, watch folders, webhooks, or custom exec scripts.

Software audit

Why this belongs in a download stack

Most media automation still polls RSS. That works for casual libraries; it fails when ratio, freeleech windows, or first-seed advantage matter. Tracker IRC channels publish the moment a torrent lands. autobrr is the listener that turns those announces into deliberate downloads instead of a later RSS scrape.

You define filters once — titles, formats, sizes, freeleech flags — and wire actions to the tools you already run. One instance can fan out to multiple remote clients and *arr apps. Built in Go with a React UI, it stays light enough for seedboxes, Docker, and bare-metal systemd hosts alike, with SQLite or Postgres, OIDC, and reverse-proxy base-path support for real deployments.

Think of it as the ops layer between “something uploaded” and “my client is already hashing.” If autodl-irssi, trackarr, and FlexGet each solved a slice of that problem, autobrr is the unified console that keeps the signal feed, the rules, and the push targets in one dark-mode dashboard.
